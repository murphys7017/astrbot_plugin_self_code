# Log Fix Enhancement Skill

## Purpose
当用户要求“检查日志并修复/增强”时，先做证据驱动诊断，再执行最小改动修复，并给出后续增强建议。

## Trigger examples
- 检查一下日志并修复
- 看看报错顺便增强一下稳定性
- 根据日志定位问题并修一下

## Input
- 用户问题描述
- 日志文件（优先）：
  - `data/logs/astrbot.log`
  - `data/logs/astrbot.trace.log`

## Workflow
1. 先确认日志文件是否存在，优先查看最近错误片段。
2. 提取关键信息：时间、模块/插件、异常类型、调用链关键节点。
3. 判断归因是否属于当前目标插件。
4. 若归因不明确或不属于当前目标插件，停止写入并返回证据与建议。
5. 若归因明确且匹配，执行最小修复（避免重构式大改）。
6. 运行基础测试，失败则回滚思路并给出替代修复方案。
7. 在修复结果后追加“增强建议”（例如：防御性校验、超时保护、日志可观测性）。

## Constraints
- 无日志证据时不得臆断根因。
- 默认不跨插件写入。
- 优先最小变更，避免引入行为回归。
- 任何修复都必须附带可验证依据。

## Output requirements
- 诊断结论（根因 + 影响范围）
- 关键证据（日志片段/文件/行号）
- 变更摘要（改了什么、为什么）
- 验证结果（测试通过/失败）
- 增强建议（最多 3 条，按优先级）
